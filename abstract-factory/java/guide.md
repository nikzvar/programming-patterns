
# Отчет по паттерну проектирования: Абстрактная фабрика (Abstract Factory)

### 1. Название и назначение паттерна

* **Название:** Абстрактная фабрика (Abstract Factory). Относится к категории *порождающих* (creational) паттернов.
* **Назначение:** Паттерн предоставляет интерфейс для создания **семейств** связанных или зависимых объектов, не привязывая клиентский код к их конкретным классам. Основная задача — инкапсулировать логику создания группы объектов, которые должны работать совместно.

### 2. Мотивация и применимость

* **Почему выбран этот паттерн:** В нашей задаче требовалось реализовать UI-инструментарий, который поддерживает несколько визуальных тем (Светлая и Темная).
* **Признаки в условии:** 1.  Наличие *семейств продуктов* (кнопки и чекбоксы существуют в вариантах Light и Dark).
2.  Жесткое требование *сочетаемости*: светлая кнопка не должна случайно оказаться рядом с темным чекбоксом. Абстрактная фабрика гарантирует, что клиент получит объекты из одного семейства.
3.  Необходимость скрыть от клиента процесс инстанцирования (клиент должен сказать «дай мне кнопку», а не писать `new DarkButton()`).

### 3. Структура взаимодействия

Взаимодействие строится на использовании интерфейсов (полиморфизме). Основные участники:

* **AbstractProduct (Абстрактные продукты):** Интерфейсы `Button` и `Checkbox`. Описывают, что умеют делать элементы.
* **ConcreteProduct (Конкретные продукты):** Классы `LightButton`, `DarkButton` и т.д. Реализуют интерфейсы продуктов. Наследуют поведение от `AbstractProduct`.
* **AbstractFactory (Абстрактная фабрика):** Интерфейс `ThemeFactory`. Объявляет методы для создания абстрактных продуктов (`createButton()`, `createCheckbox()`).
* **ConcreteFactory (Конкретные фабрики):** Классы `LightFactory` и `DarkFactory`. Реализуют методы фабрики, возвращая конкретные экземпляры продуктов.
* **Client (Клиент):** Класс `Application`. Работает с фабриками и продуктами **только через их интерфейсы**. Связь клиента с фабрикой — это композиция/агрегация (клиент хранит ссылку на `ThemeFactory`).

### 4. Результаты применения

* **Что мы выиграли (Плюсы):**
* *Гарантия сочетаемости:* Клиент всегда получает продукты из одной темы.
* *Изоляция конкретных классов:* Клиентский код `Application` ничего не знает о `DarkButton`.
* *Легкость добавления новых семейств:* Чтобы добавить `NeonTheme`, нам нужно создать `NeonFactory` и неоновые продукты. Старый код клиента и других фабрик вообще не изменится (Соблюдение принципа Open/Closed).


* **Недостатки и ограничения (Минусы):**
* *Сложность добавления новых ТИПОВ продуктов:* Если мы захотим добавить в UI новый элемент, например `TextField` (Текстовое поле), нам придется менять интерфейс `ThemeFactory` и **все** существующие конкретные фабрики (LightFactory, DarkFactory и т.д.). Это нарушает принцип Open/Closed на уровне интерфейса фабрики.



### 5. Демонстрация реализации (кода)

В написанном нами Java-коде логика работает так:

1. В `Main` (точке сборки) мы определяем конфигурацию среды (выбираем тему).
2. В зависимости от условия создается конкретная фабрика: `ThemeFactory factory = new DarkFactory();`.
3. Эта фабрика передается в `Application` (внедрение зависимостей).
4. `Application` вызывает `factory.createButton()`. В этот момент происходит позднее связывание: JVM знает, что под маской `ThemeFactory` скрывается `DarkFactory`, и возвращает `DarkButton`.
5. Вызов `button.paint()` выводит отрисовку темной темы.

* **Примеры в Open Source и стандартных библиотеках:**
* В Java SDK: `javax.xml.parsers.DocumentBuilderFactory` — абстрактная фабрика, скрывающая создание конкретных парсеров XML в зависимости от реализации (Xerces, стандартный парсер JDK и т.д.).
* Паттерн часто применяется в GUI-фреймворках кроссплатформенных ОС для отрисовки нативных окон под Windows, macOS и Linux.



### 6. Родственные паттерны

* **Отличие от Фабричного метода (Factory Method):** Фабричный метод создает *один* продукт и использует для этого механизм наследования (подклассы переопределяют метод создания). Абстрактная фабрика создает *семейства* продуктов и использует композицию (клиент делегирует создание объекту-фабрике). Часто методы внутри Абстрактной фабрики реализуются именно как Фабричные методы.
* **Отличие от Строителя (Builder):** Строитель фокусируется на пошаговом создании *одного сложного* объекта (как мы собирали пользователя с опциональными полями). Абстрактная фабрика делает акцент на создании *сразу нескольких связанных* объектов.
* **Связь с Одиночкой (Singleton):** Конкретные фабрики (`LightFactory`, `DarkFactory`) обычно не имеют состояния (у них нет полей данных). Поэтому в реальных проектах их чаще всего реализуют как **Singleton**, чтобы не плодить лишние объекты в памяти при каждом запросе интерфейса.
